## Vocabulary drift

### V1: PRD "Search ViewModel" / "Weather ViewModel" vs Spec/Plan `SearchViewModel` / `WeatherViewModel` / `MainViewModel`
**PRD names:** "Search ViewModel", "Weather ViewModel" (spaced, no shell ViewModel).
**Spec/Plan use:** `SearchViewModel`, `WeatherViewModel`, plus a new `MainViewModel` shell
that mediates the activation handoff (Approach A).
**Note:** These are implementation constructs, not domain glossary terms (the glossary is
"definitions only — no implementation"), so the absence of `MainViewModel` from `Context.MD`
is expected, not drift. The PRD-vs-Spec naming is a cosmetic spacing difference and the
shell ViewModel is a legitimate design elaboration. Non-gating observation.
**Suggested resolution:** none required; optionally align PRD module names to the code names.

## Coverage gaps

### G1: Technical-Context Instrumentation requires logging every outbound Open-Meteo call; F1 Spec/Plan implement none
**Upstream:** `Technical-Context.MD` › Instrumentation — "Every outbound Open-Meteo call is
logged with its outcome (request target + success/failure + status)."
**Downstream:** Neither the F1 Spec's data flow / error handling nor the Plan's
`OpenMeteoGeocoder` / `OpenMeteoWeatherProvider` (constructed with `HttpClient` only, no
`ILogger`) logs outbound calls. The generic host wires `Microsoft.Extensions.Logging` but the
clients never use it.
**Weight:** the Instrumentation section is explicitly marked an *inferred default* (not an
Overriding Principle), and the Plan's Context-references block scopes only Principle #1
(no secrets) as binding. So this is a soft default, not a hard contract — a gap to surface,
not a blocker. Worth a one-line decision: either add a logging story/line to F1 or record
that per-call logging is deferred.
